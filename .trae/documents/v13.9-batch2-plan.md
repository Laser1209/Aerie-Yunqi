## 三、Batch 2.2: 回复校验机制（准确性 Guard + 质量 Judge）

### 3.1 架构设计

```
用户消息 → Pipeline → LLM 生成回复 → [Guard 层] → [Judge 层] → 输出
```

### 3.2 Accuracy Guard（准确性守卫）

**职责**: 校验回复内容的事实准确性和安全合规性

**检查项**:
- 事实一致性：回复内容是否与已知事实/上下文矛盾
- 安全合规：是否包含敏感/违规内容
- 工具调用结果：如果涉及工具调用，回复是否准确反映工具返回结果
- 数值准确性：涉及数字/日期/时间的内容是否合理

**实现方式**:
- 轻量级关键词 + 规则检查（快速路径，10ms 内完成）
- 对高风险内容触发 LLM 二次校验（慢速路径）
- 不通过时：触发重写（最多 2 次），或附加风险提示

### 3.3 Quality Judge（质量评判）

**职责**: 评估回复质量，确保符合人设和场景要求

**评估维度**:
- **信息量**: 是否充分回答了问题，有无冗余或不足
- **语气一致性**: 是否符合当前人设和关系层级
- **长度适宜**: 既不过短也不过长
- **情感价值**: 是否传递了恰当的情绪温度
- **场景适配**: 办公模式 vs 聊天模式的语气切换

**实现方式**:
- 基于规则的快速评估（长度检查、关键词匹配）
- 办公模式下额外检查专业度和结构清晰度

### 3.4 集成位置

在 `pipeline.py` 的 `_generate_reply()` 之后、`_post_process_reply()` 之前插入校验层：

```python
# pipeline.py 新增
async def _validate_reply(
    self,
    reply_text: str,
    msg: IncomingMessage,
    context: dict,
    route_mode: str,
) -> tuple[str, dict]:
    """回复校验：准确性 Guard + 质量 Judge。"""
    validation = {
        "guard_passed": True,
        "judge_score": 1.0,
        "rewrite_count": 0,
        "issues": [],
    }
    # 1. Accuracy Guard
    # 2. Quality Judge
    # 3. 必要时触发重写
    return reply_text, validation
```

## 四、Batch 2.3: 主动消息推送（定时/情绪/事件三类触发）

### 4.1 三类触发源

| 触发类型 | 触发时机 | 示例 |
|---------|---------|------|
| **定时触发 (Cron)** | 固定时间点 | 早安问候、日程提醒、每日简报 |
| **情绪触发 (Emotion)** | 用户情绪达到阈值 | 用户低落时主动关心、用户开心时分享喜悦 |
| **事件触发 (Event)** | 特定系统事件 | 长时间未对话、纪念日、新功能上线 |

### 4.2 架构设计

```
┌─────────────────────────────────────────────────┐
│              Push Engine (主动推送引擎)           │
├──────────┬──────────┬───────────────────────────┤
│ CronScheduler │ EmotionDetector │ EventBus     │
└────┬─────┴────┬─────┴───────┬───────────────────┘
     │          │             │
     └──────────┼─────────────┘
                ▼
         Push Planner (生成推送内容)
                │
                ▼
         Rate Limiter (频率控制)
                │
                ▼
         QQ / WebUI (推送分发)
```

### 4.3 核心功能

- **推送内容生成**: 基于当前人设 + 用户状态 + 场景生成自然的主动消息
- **频率控制**: 避免过度打扰，每日上限、静默时段
- **用户画像感知**: 根据用户习惯调整推送时机
- **可配置**: 前端可开关各类推送、调整频率

### 4.4 涉及文件

- 新增/完善: `core/push_engine.py`
- 修改: `core/companion.py`（初始化推送引擎）
- 修改: `core/api_server.py`（推送管理 API）
- 完善: `electron/src/renderer/js/proactive-manager.js`（前端管理面板）

## 八、风险与注意事项

| 风险 | 影响 | 应对方案 |
|------|------|---------|
| 图标替换导致布局偏移 | 中 | 逐一检查各页面布局，微调 CSS |
| 回复校验增加延迟 | 中 | 快速路径（规则）优先，慢速路径（LLM）仅高风险触发 |
| 主动推送过于频繁 | 高 | 默认保守配置，每日上限 3-5 条，静默时段 23:00-8:00 |
| 任务规划过度消耗 Token | 中 | 设置规划步数上限（最多 10 步），简单任务不触发规划 |
| 图标数量增加页面体积 | 低 | SVG 图标体积小（每个 <1KB），30 个新增 < 30KB |
