# Aerie 二期需求追踪矩阵

## 1. 文档目的与裁决口径

本文件把二期主计划的目标、现状缺口、工作包、八层产品架构和强制工程合同，追踪到一期 Phase/Task/ADR、当前代码、当前测试、二期批次、证据类型和阶段门。它证明的是“需求有明确去向”，不把文档存在、一期 `Done`、测试文件名或历史测试数字误算为二期功能验收。

发生冲突时严格采用以下权威顺序：

1. `documents/Level_up/AI_Vibe_Coding/` 的主控计划、accepted ADR、核心合同和全局验收文件。
2. Phase 00-15、Task 00-15 及已脱敏 Evidence。
3. 当前代码、当前测试和可复现实验；当前事实优先于历史计划判断。
4. `documents/Ita/Ita_Aerie_Companion_Spec.md` 的产品愿景和八层职责。
5. `documents/Level_up/` 的原始方案需求池。
6. `documents/Level_up/实施计划.md` 的实施前快照。

一期 Phase/Task 的 `done/completed` 只表示对应合同基线曾完成。二期仍需完成激活、整合、持久化、常驻化、独立进程化、产品化和发布硬化，不能由一期状态自动继承二期验收。

## 2. 追踪模型与覆盖率

### 2.1 状态定义

| 状态 | 含义 |
|---|---|
| `Baseline` | 一期合同在当前代码中存在，且冻结基线的直接测试覆盖该合同；不代表二期工作包完成 |
| `Partial` | 已有一期基础，但二期语义、集成或验收证据不完整 |
| `Missing` | 当前代码、测试或运行能力尚不存在 |
| `Contradicted` | 当前代码或当前可复现实验直接推翻历史 Pass/计划判断 |
| `Blocked` | 追踪关系完整，但前置阶段门或阻断项不允许实施/验收 |
| `Review` | 文档内容已完成自检，等待 Reviewer/PO 裁决 |

### 2.2 双向追踪规则

正向链路固定为：

```text
需求 ID -> 权威来源 -> 一期 Phase/Task/ADR -> 当前代码 -> 当前测试/证据
        -> 唯一主责 P2 批次 -> 依赖批次 -> 验收门 -> 最终 Evidence
```

反向链路固定为：

```text
代码/测试/迁移/API/IPC/ADR/Evidence -> 需求 ID -> 主责批次 -> 阶段门
```

强制规则：

1. 每个原子需求只能有一个 primary owner；其他工作包只列依赖，禁止重复计数。
2. 修改代码或共享合同时，PR/提交必须引用需求 ID、批次 ID、证据类型和受影响阶段门。
3. 新测试必须反向引用至少一个需求 ID；没有需求 ID 的测试不能单独证明追踪完整。
4. 文件存在、测试名称存在和历史测试数字只证明关联，不自动证明行为已验收。
5. `Not Run`、`Captured`、`Pass with limitations`、历史 Evidence 和未复现人工描述不计作当前完整验证。
6. 负向合同必须验证数据库、网络、ACK、文件、通知、QQ、模型外呼和后台任务的适用副作用；只验证返回值不足以证明零副作用。
7. 阶段门采用布尔 AND；覆盖率不能抵消任一 Fail、Not Run、未批准例外或缺失证据。
8. 复合顶层需求在实施批次中继续拆成 `.1/.2/...` 原子验收行；冻结后的原子行总数才是功能验收覆盖率分母。
