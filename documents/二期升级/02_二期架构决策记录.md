# Aerie 二期架构决策记录

## 1. 文档状态与使用规则

### 1.1 状态定义

| 状态 | 含义 | 是否允许据此实施 |
|---|---|---|
| `Accepted (Inherited)` | 一期权威 ADR 或控制合同已经接受；本文件中的编号只是二期索引别名，不构成第二次裁决 | 可以遵守既有边界，但不得借二期改变其含义 |
| `Proposed for G0` | 二期候选决定，等待 ARCH 复核与 PO 在 G0 明确批准 | 不允许据此修改生产代码、数据、配置或构建 |
| `Superseded` | 后续有明确的新 ADR 替代，且保留原决定和迁移记录 | 仅按替代 ADR 的生效范围实施 |

### 1.2 权威顺序

发生冲突时按以下顺序裁决：

1. `documents/Level_up/AI_Vibe_Coding/` 中已经接受的 ADR、核心合同和主控计划。
2. `Aerie_项目二期升级更新计划.md` 的不可变合同与 G0/G1 候选边界。
3. 当前代码与可复现测试证据；代码用于说明现状，不能自动推翻已接受合同。
4. 本文件中 `Accepted (Inherited)` 的索引说明。
5. 本文件中 `Proposed for G0` 的候选决定；未获 PO 批准前不得作为实施依据。

如需改变一期已接受决定，必须新增明确标注 `supersedes` 的 ADR，说明数据迁移、兼容期、回滚和用户影响，并重新经过 PO 批准；不得直接修改索引别名来制造“已批准”状态。

## 3. 当前代码不等于目标实现

Phase 00-15 的 `done/completed` 表示一期合同基线和对应测试曾完成，不表示二期产品化目标已经实现。G0 审查必须保留以下现状差距：

| 领域 | 当前代码事实 | 二期目标，尚未实现 |
|---|---|---|
| 对话组合 | `core/companion.py` 组合一套 `Pipeline` 与 `ContextBuilder` | 生命周期和共享依赖进一步显式化，但不创建第二主链 |
| Persona | `persona_hub_source_v1` 开启时从 Persona Hub 生成旧结构；关闭时仍走旧 YAML 兼容路径 | Persona Hub 成为稳定唯一写入入口，旧 YAML 只由投影生成 |
| 会话数据 | Pipeline 写 `chat_log` 并按 Flag 写 Canonical；历史 API 仍主要读 `chat_log` | Canonical 成为主读模型，兼容期持续双写核对 |
| Core SSE | `core/event_stream.py` 使用进程内有限 replay window | 后续可增加 Core 自己的持久 cursor/outbox，但不得复用 World 协议或数据库 |
| World Sidecar | `core/world_port.py` 在 Core 进程内构造 `LocalWorldSidecarService`，`RemoteWorldAdapter` 仍为 duck-typed 对象调用 | 独立进程、认证 transport、版本握手、真实 spawn/stop/restart |
| Supervisor | `electron/src/plugin-supervisor.js` 只记录 register、heartbeat、crash 和 fuse | 拥有子进程、退避、优雅停止、残留清理和恢复 |
| 运行配置 | Python `FeatureFlags` 读取 YAML 快照与环境覆盖；Electron World Host 只读环境变量 | Python/Electron 消费同一脱敏 effective snapshot |
| 构建 | `package.json` 与 `electron-builder.yml` 同时包含构建配置；当前过滤包含 `data/**` 与 `.venv/**`，安装权限字段互相冲突 | 单一构建配置、allowlist、固定运行时、明确安装范围 |

因此，文档中的“目标决定”不能作为“代码已完成”的证据；每项实现仍须进入其影响批次，完成 Red、最小实现、关联测试、全量回归和回滚验证。

## 5. 共享合同串行规则

1. 共享合同包括 WorldPort DTO、EventEnvelope、Canonical schema、Persona 投影、Runtime Config schema、Sidecar handshake、IPC schema 和构建清单。
2. 同一时刻只允许一个批次修改同一共享合同；后续消费者批次必须依赖该合同批次的已接受 commit。
3. 修改共享合同前先更新 ADR 和失败的契约测试，再修改 provider，最后修改 consumer；禁止先让两端各自猜测字段。
4. 新字段只追加，不复用旧字段表达新含义；新枚举必须有 `unknown/unsupported` 路径。
5. 数据与状态合同必须携带 owner、revision、idempotency key 或 sequence 中适用的字段，并明确终态不可逆规则。
6. 每个合同变更都必须验证 Flag off、旧 API/Adapter、数据守恒和回滚；Evidence 只记录脱敏 ID、hash、计数、耗时和结论。
7. 任何合同未获批准、Reviewer 不一致或回滚不可复现时，依赖批次保持 `Blocked`。

## 6. 继承决定

### ADR-P2-001：单一 Companion、Pipeline 与 ContextBuilder 主链

| 字段 | 内容 |
|---|---|
| 状态 | `Accepted (Inherited)` |
| 索引性质 | 二期索引别名，不是第二次裁决 |
| 来源 | [六方案冲突裁决](../Level_up/AI_Vibe_Coding/01_六方案冲突裁决.md) 与二期主计划工程不可变合同 1 |
| Owner / Reviewer | CORE / ARCH、QA |
| 批准要求 | 已继承；若改变必须新建 superseding ADR 并经 PO 批准 |
| 影响批次 | P2-02、P2-03、P2-04、P2-05、P2-10、P2-11 |

**上下文。** 当前 `Companion` 构造一套生产 `Pipeline`，并向其中注入一个 `ContextBuilder`、共享 Repository 和领域引擎。二期需要增加配置、摘要、流式、会话与知识能力，但整体替换会造成身份、持久化、副作用和旧 API 分裂。

**决定。** `Companion` 继续作为 Core 组合根；生产对话只演进现有 `Pipeline` 与 `ContextBuilder`。新增能力必须作为现有主链的子组件、Repository、Service、Port、Adapter 或 hook 接入。禁止创建长期并行的 `Pipeline v2`、`ContextBuilder v2` 或第二个生产组合根。

**兼容与回滚。** 保留 `Pipeline.handle()`、旧同步入口、旧 Context 行为和现有 API；新增子能力分别受 Flag 控制，回滚只关闭对应 Flag 或移除 hook，不替换主链、不删除新数据。

**验证与证据。** 静态守护生产代码中 Pipeline/Builder 定义与组合入口；组合测试证明共享实例唯一；每个新增 Flag 验证关闭时不调用新 handler、不写新数据、不产生外部副作用；Evidence 指向受影响测试与 commit。

**上下文。** 当前 Flag 开启时 `config/persona_loader.py` 从 Persona Hub 读取并投影旧结构；Flag 关闭时保留旧 YAML 读取和写入路径。该旧路径是回退能力，不代表 YAML 与 Hub 可以长期互为真源。

**决定。** Persona Hub 是 Persona profile 的唯一写入真源；Legacy Projector 只从 Hub 单向生成兼容结构。旧 Persona YAML 在切换稳定期仅作为只读兼容投影。`persona_behavior.yaml` 暂时承担行为参数兼容配置，但不得覆盖 Persona 身份、价值观和所有权语义。

**兼容与回滚。** `persona_hub_source_v1` 关闭时允许回到旧读取路径；回滚不反向导入或覆盖 Hub。旧 YAML API 在兼容期保留读取，Hub 启用时继续拒绝直接写 Persona YAML。

**验证与证据。** 覆盖 Hub CRUD、active persona、投影字段/hash、旧 API 只读、Flag off 旧路径、Actor/Persona owner 隔离和重启一致性；投影差异必须可审计且不得包含秘密。

| 来源 | [ADR：桌面 QQ 短期隔离长期共享](../Level_up/AI_Vibe_Coding/decisions/ADR-桌面QQ短期隔离长期共享.md) |

**上下文。** Desktop 与 QQ 具有不同的交互节奏和短期上下文，但绑定同一用户时需要长期关系和记忆连续。仅按 `user_id` 或仅按 Channel 建模都会造成串线或割裂。

**验证与证据。** Desktop/QQ 并发、多 Actor、多 Persona、错误绑定、解绑、重启和导出测试必须证明短期不串线、长期共享符合绑定关系。

### ADR-P2-005：Core 与 Sidecar 数据库单写所有权

| 字段 | 内容 |
|---|---|
| 状态 | `Accepted (Inherited)` |
| 索引性质 | 二期索引别名，不是第二次裁决 |
| 来源 | [数据所有权与迁移纪律](../Level_up/AI_Vibe_Coding/03_数据所有权与迁移纪律.md) |
| Owner / Reviewer | CORE、WORLD / ARCH、QA |

| 影响批次 | P2-01、P2-04、P2-06、P2-07、P2-08、P2-10、P2-12 |

**上下文。** Core 与 World 有不同生命周期、可靠性和回滚要求。共享连接、跨库 JOIN 或同一实体双写会让 Sidecar 隔离、恢复和降级失效。

**决定。** `aerie.db` 只由 Core 写；`world.db` 只由 Sidecar 写。双方只通过版本化 DTO 和 WorldPort/transport 交换信息，禁止跨库 JOIN、共享 SQLite connection、复制整表或同一实体双写两库。Core 可保存候选处理的 Core 终态，但不得修改 World Outbox/ACK 表。

**兼容与回滚。** Remote 失败时按 `Remote -> InProcess -> Null` 降级；保留 `world.db`、Outbox 和 checkpoint。应用回退使用兼容 schema，不执行破坏性 down migration。

**验证与证据。** 静态检查 import/SQL 路径，运行时使用独立临时数据库，分别验证迁移、写入、文件句柄、`quick_check`、故障恢复和无跨库访问。

### ADR-P2-006：Core SSE 与 World 可靠事件协议分离

| 字段 | 内容 |
|---|---|
| 状态 | `Accepted (Inherited)` |
| 索引性质 | 二期索引别名，不是第二次裁决 |
| 来源 | [ADR：Core SSE 与 World 可靠协议分离](../Level_up/AI_Vibe_Coding/decisions/ADR-Core%20SSE与World可靠协议分离.md) |
| Owner / Reviewer | CORE、WORLD / ARCH、QA |
| 批准要求 | 已继承；禁止把两个协议或数据库合并 |
| 影响批次 | P2-04-B03、P2-07-B03、P2-08-B01、P2-09-B02、P2-11-B01 |

**上下文。** UI 事件优先低延迟和可恢复渲染；World 事件必须跨进程重启后至少一次投递并在副作用终态后确认，两者可靠性语义不同。

**决定。** Core UI 继续使用 EventEnvelope、SSE、`Last-Event-ID`、sequence 与 Renderer 去重。World 继续使用独立的 Outbox、ACK cursor、checkpoint、heartbeat 和幂等键。二期若为 Core 增加持久事件恢复，仍使用 Core 自己的存储和语义，不复用 `world.db` 或 World ACK。

**兼容与回滚。** `chat_stream_v1` 或后续流式 Flag 关闭时回到完整响应/poll；World transport 失败时回退 Adapter，未 ACK 事件保留重放。任何一边失败均不得阻断旧文字聊天。

**验证与证据。** 分别覆盖 SSE 重连/超窗/去重与 World 重放/乱序/重复/ACK 丢失/Core 重启/Sidecar 重启；不得用 SSE 通过证明 World 可靠性，反之亦然。

### ADR-P2-007：WorldPort 渐进 Adapter 与故障回退

| 字段 | 内容 |
|---|---|
| 状态 | `Accepted (Inherited)` |
| 索引性质 | 二期索引别名，不是第二次裁决 |
| 来源 | [ADR：WorldPort 渐进 Sidecar](../Level_up/AI_Vibe_Coding/decisions/ADR-WorldPort渐进Sidecar.md) |
| Owner / Reviewer | ARCH、WORLD / CORE、QA |
| 批准要求 | 已继承；不得让业务层或 Renderer 绕过 WorldPort |
| 影响批次 | P2-07、P2-08、P2-09 |

**上下文。** 一期已建立稳定 DTO、Null/InProcess/Remote Adapter 和确定性世界基线，但当前 Remote 仍是同进程对象适配。二期需要替换 transport，而不能改变所有 Core 调用点。

**决定。** Core 只依赖 WorldPort 的版本化 DTO 与方法语义；先用 InProcess 验证世界领域与 Scheduler，再将 Remote Adapter 的 client 替换为真实 transport。Null 与 InProcess 是正式降级路径，不是临时测试 stub。

**兼容与回滚。** `world_sidecar_v1` 关闭或握手失败时切换 InProcess；InProcess 不可用或总开关关闭时切 Null。回滚不得改变聊天 Pipeline、删除世界数据或伪造 healthy。

**验证与证据。** 对三种 Adapter 运行同一契约测试；验证协议不兼容、认证失败、端口占用、崩溃、暂停、恢复和原子切换；Dashboard 必须显示真实 adapter 与降级原因。

### ADR-P2-008：World 候选与 Core 副作用终态

| 字段 | 内容 |
|---|---|
| 状态 | `Accepted (Inherited)` |
| 索引性质 | 二期索引别名，不是第二次裁决 |
| 来源 | [ADR：图片副作用归 Core](../Level_up/AI_Vibe_Coding/decisions/ADR-图片副作用归Core.md) |
| Owner / Reviewer | WORLD、CORE / ARCH、SEC、QA |
| 批准要求 | 已继承；World 不得获得副作用权限 |
| 影响批次 | P2-05、P2-06、P2-07-B03、P2-09-B02 |

**上下文。** World 可以持续生成模拟事件和建议，但它不拥有用户权限、Provider、QQ、通知、文件或图片资产。At-least-once 重放还要求副作用具备幂等终态。

**决定。** World 只发布结构化、脱敏、有 owner 与 idempotency key 的候选。Core 执行 Policy/Judge、预算、生成、审核、持久化和 Delivery；只有 Core 已持久化终态后才推进 World ACK。`disabled/offline/retryable` 等非终态不得 ACK。

**兼容与回滚。** 关闭候选 Consumer 时保留候选且不消费、不 ACK；图片或 QQ 不可用时降级文本或保持可重试，不阻断聊天。

**验证与证据。** 静态证明 `world_service` 不导入 QQ、图片 Provider 或系统工具；重复候选、重启、ACK 丢失和执行中断不能产生重复消息、文件、费用或通知；终态记录与 ACK cursor 可双向追踪。

## 7. 二期候选决定

> 本节四项全部保持 `Proposed for G0`。只有 PO 在 G0 留下明确批准记录后，ARCH 才能把对应状态更新为 `Accepted`，并将直接影响批次置为可执行。不得以本文创建、Git 提交或 Review 状态推定批准。

### ADR-P2-009：统一 RuntimeConfigService 与有效配置快照

| 字段 | 内容 |
|---|---|
| 状态 | `Proposed for G0` |
| Owner / Reviewer | CORE / DESK、SEC、QA、PO |
| 批准要求 | ARCH 确认 schema 与兼容边界；SEC 确认秘密脱敏；PO 在 G0 明确接受 |
| 影响批次 | P2-01-B01 至 P2-01-B04、P2-02-B01、P2-11-B01 |
| Feature Flag | `runtime_control_v1`，初始默认 off |

**上下文。** Python `FeatureFlags` 在实例化时读取 `config/settings.yaml`，查询时允许环境变量覆盖；Electron World Host 只读环境变量。两端可能显示不同值，现有热加载也不会原子重建 WorldPort 或后台任务。

**候选决定。** 在 Core 增加 `RuntimeConfigService`，作为非秘密运行配置和 Feature Flag 的 effective snapshot 权威。优先级固定为：进程启动环境变量只读覆盖 > 经校验的版本化本机持久配置 > 仓库默认值。快照至少返回 `key/effectiveValue/source/revision/mutable/requiresRestart/dependencies/validationErrors`；秘密只返回 `configured` 与来源，不回显值。现有 `FeatureFlags.is_enabled()` 保持 facade，并逐步委托同一快照；Electron 不再自行解释 YAML 或环境变量，只消费 Core 的脱敏结果。

**兼容。** 保留现有 `FeatureFlags` 调用签名、`config/settings.yaml` 和旧 YAML API。只有声明为 dynamic 的键允许热应用；其余返回 `requiresRestart=true`，不得伪装已生效。

**回滚。** 关闭 `runtime_control_v1` 回到当前 YAML+env/legacy Electron 读取；恢复上一配置 revision 或原子备份。回滚不删除审计记录，也不改变环境变量覆盖。

**所需证据。** Python/Electron 对全部白名单键的值、来源、revision 和重启要求 100% 一致；expected revision 冲突不覆盖；原子写失败可恢复；依赖环与非法值稳定拒绝；秘密扫描和 API 快照无值泄漏；Flag off 零新增数据库、网络、文件和后台任务副作用。

### ADR-P2-010：Desired、Actual、Adapter 与界面可见性的所有权分离

| 字段 | 内容 |
|---|---|
| 状态 | `Proposed for G0` |
| Owner / Reviewer | ARCH、CORE、DESK / QA、SEC、PO |
| 批准要求 | CORE/DESK 对状态机和命令语义共同签字；PO 在 G0 明确接受 |
| 影响批次 | P2-01-B04、P2-07、P2-08、P2-09 |
| Feature Flag | `runtime_control_v1`；Dashboard 控制另受 `world_dashboard_control_v1` 约束 |

**上下文。** 当前 Dashboard 的 show/hide 只改变面板可见性，Supervisor 只记录状态，无法表达用户期望、真实进程、Core Adapter 和模拟暂停之间的差异。

**候选决定。** Core Runtime Config 持有 `desired`；Electron Plugin Supervisor 持有 Sidecar 子进程 `actual`；Core `WorldRuntimeManager` 持有当前 Adapter、连接健康与 effective runtime；Renderer 只通过窄 IPC 发出命令并展示脱敏快照。`enable/disable`、`start/stop`、`pause/resume`、`restart`、`show/hide` 是不同操作。状态写命令必须带 idempotency key 与 expected revision，响应至少包含 accepted/rejected、desired、actual、revision、fallback adapter 和稳定 error code。

**兼容。** 现有 show/hide 保留为纯可见性操作；后端缺少新接口时 UI 显示 `unsupported` 并回到只读状态，不无限重试。文字聊天不依赖 World 实际状态。

**回滚。** 关闭控制 Flag 后停止接收新控制命令，保留只读快照并按 `Remote -> InProcess -> Null` 降级；恢复上一 desired revision。不得通过隐藏 Dashboard 冒充停止世界。

**所需证据。** 状态机每条转换、重复命令、revision 冲突、进程崩溃、心跳超时、pause 查询、应用重启和 UI 隐藏均有测试；状态变化三秒内传播；任何失败不阻断聊天且不产生残留进程或虚假 healthy。

### ADR-P2-011：独立 Sidecar 的本机认证 Transport 与进程所有权

| 字段 | 内容 |
|---|---|
| 状态 | `Proposed for G0` |
| Owner / Reviewer | WORLD、DESK / ARCH、SEC、QA、PO |
| 批准要求 | ARCH 接受协议；SEC 接受认证与令牌生命周期；PO 在 G0 明确接受 |
| 影响批次 | P2-07-B01 至 P2-07-B03、P2-08-B01 至 P2-08-B03、P2-09-B01 |
| Feature Flag | `world_process_supervision_v1`，依赖 `runtime_control_v1`；用户总开关继续为 `world_sidecar_v1` |

**上下文。** 当前 `RemoteWorldAdapter` 接受 duck-typed client，Core 直接构造 `LocalWorldSidecarService`；Electron Supervisor 不拥有真实 World 子进程。因此已有协议与存储测试不能证明进程隔离、认证、端口、停止和恢复成立。

**候选决定。** 使用仅绑定 `127.0.0.1` 的 HTTP/JSON 本机 transport，Sidecar 启动时选择动态端口，并使用每次启动重新生成的短期 bearer token。令牌通过受控进程环境或等价的继承通道传递，不写 argv、磁盘、日志或 Renderer。Electron Supervisor 唯一负责 spawn、handshake、heartbeat、stop、restart、退避、熔断和残留进程清理；Core `RemoteWorldAdapter` 作为客户端。握手必须声明 protocol/version range、service version、instance ID、startup epoch 和 capabilities；不兼容或认证失败时稳定拒绝并降级。Renderer 永远不得获得 endpoint 或 token。

World 可靠性仍由 Outbox、replay、ACK cursor 与 checkpoint 提供；HTTP 只是 transport，不得用 HTTP 成功响应替代终态 ACK。

**兼容。** 保持 WorldPort 五方法及现有 DTO 语义；测试可继续以 Local Service 作为 contract fixture。握手或 transport 不可用时回到 InProcess/Null，聊天和旧主动路径继续工作。

**回滚。** 关闭 `world_process_supervision_v1`，优雅停止子进程并清除短期 token/endpoint，切换 InProcess；超时后受控强制结束。保留 `world.db`、checkpoint、Outbox 和未 ACK 事件。

**所需证据。** 真实 spawn/stop/restart、动态端口占用、非 loopback 拒绝、无 token/过期 token/错误 token、协议不兼容、心跳丢失、连续崩溃熔断、Core/Sidecar 分别重启、ACK 重放和无残留 PID/端口/文件锁全部通过；日志和进程启动记录不得含 token。

### ADR-P2-012：单一版本与构建配置、包内容 Allowlist

| 字段 | 内容 |
|---|---|
| 状态 | `Proposed for G0` |
| Owner / Reviewer | REL / SEC、QA、PO |
| 批准要求 | REL 提供构建清单；SEC 接受包内容；PO 明确选择安装范围并在 G0 接受 |
| 影响批次 | P2-11-B04、P2-12-B01 至 P2-12-B06 |
| Feature Flag | 无；属于发布硬门禁，不得由普通产品 Flag 绕过 |

**上下文。** 当前 `electron/package.json` 与 `electron/electron-builder.yml` 同时包含构建配置；YAML 过滤仍包含 `data/**` 和开发 `.venv/**`；`requestedExecutionLevel: requireAdministrator` 与 `perMachine:false` 不一致。双配置和黑名单式过滤无法证明最终包不夹带本机数据。

**候选决定。** `electron/package.json.version` 是产品版本唯一真源；`electron-builder.yml` 是 Electron Builder 唯一打包配置，构建脚本显式指定该文件，获批实施时删除 `package.json` 中重复的 `build` 块。产物使用 allowlist，仅包含明确应用源码、静态资源、公共默认配置、经过锁定和验证的 Python runtime/依赖以及经扫描的 `.env.example` 模板；明确排除真实 `.env`、开发 `.venv`、`data/`、`logs/`、`uploads/`、缓存、备份、测试、Evidence、NapCat 私有目录和用户路径。

安装范围建议采用与现有 `perMachine:false` 一致的 per-user 策略，并使用非管理员执行级别；该选择在 PO 明确批准前仍未冻结。若 PO 选择 per-machine，必须同步调整权限、数据目录、升级/卸载和 UAC 测试，不能保留冲突字段。

**兼容。** 保留 electron-builder、NSIS 与 Portable 目标；用户数据继续位于独立 userData，不随程序覆盖或卸载默认删除。旧版可读取新 schema 的兼容字段，版本回退不执行 down migration。

**回滚。** 停止发布候选并恢复上一份已验证的构建配置或上一签名安装包；保留用户数据与 schema。任何禁入文件或高危依赖命中都阻断发布，不允许通过排除扫描结果继续构建。

**所需证据。** package/About/安装器/Portable/manifest/Release 版本一致；展开 asar 与 extraResources 后禁入项为 0；secret、依赖、许可证和 SBOM 检查通过；固定工具链可重复生成清单；全新安装、覆盖升级、程序回退、卸载保留数据、快捷方式、自动启动和 SHA-256/签名验证全部通过。

## 9. G0 与 P2-01 边界

本文档保持 `gate_status: open`，原因如下：

1. `ADR-P2-009` 至 `ADR-P2-012` 仍为候选，尚无 PO 明确接受记录。
2. B02 已记录的测试、安全、依赖和打包失败必须分别修复、约束或形成经批准的风险处置；创建 ADR 不会把失败改成通过。
3. G0 还要求 13 个工作包追踪率 100%、R-01 至 R-17 责任人与触发操作完整、基线可复现以及 PO 批准进入下一阶段。
4. 在 G0 关闭前，不得开始 `P2-01` 功能实现，不得修改 RuntimeConfig、Sidecar transport、生命周期控制或构建配置来提前落实候选 ADR。
5. G0 通过后，只能将编排文件明确允许的首批任务置为 Ready；后续阶段仍受各自依赖和门禁约束。
