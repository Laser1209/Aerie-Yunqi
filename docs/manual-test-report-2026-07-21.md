### 4.12 灵动岛原生窗口

- 实际作用：顶部约 200x43 胶囊；单击后展开为约 320x534 的快捷操作、媒体、陪伴、通知和系统状态面板。
- 根因：Electron 28 在 Windows 上使用 `"floating"` 层级时未获得原生 `WS_EX_TOPMOST`，主窗口前置或最大化后会遮挡灵动岛。
- 修复：Windows 改用 `"screen-saver"` 层级，并在主窗口 show、focus、restore、maximize、unmaximize、原生/HTML 全屏切换和统一唤回路径中重新断言置顶。
- 运行态证据：17:43 冷启动后的主进程 PID `21008`；主窗口最大化时灵动岛 HWND `0xF009A`，`WS_EX_TOPMOST=true`、Z-order `2`；主窗口 HWND `0xC509EE`，非 topmost、Z-order `6`，前台焦点仍属于主窗口。两个 renderer 命令行均包含 `--enable-sandbox`。
- 视觉/交互：胶囊文字、头像和图标均可见；单击可展开，快捷操作、媒体、陪伴、通知和系统状态完整显示；关闭后恢复约 200x43 胶囊态。
- 结果：主窗口显示、聚焦、最大化、还原、F11 全屏和二次启动唤回后灵动岛持续可见且置顶，本问题通过复测。
- 导航复测：日报使用专用 `brief:show` 协议，`calendar` 统一映射到 `memorial`；托盘日报、完整日报、设置和启动日报均经过 readiness-aware 队列，发送失败会保留待处理动作。

### 6.3 资源热点复测

- 灵动岛媒体状态已改为自适应轮询：展开或播放时 5 秒，折叠且未播放时 15 秒，隐藏时暂停。
- 真实计时 20 秒内仅观察到 1 个由 Electron 主进程创建的 PowerShell/WinRT 子进程；查询与控制均为单飞、隐藏窗口且有超时终止。
- 长期仍建议把短进程方案替换成持久 WinRT helper，但当前频率和稳态 CPU 已达到本轮门槛。
- HTTP 连接没有充分复用，端口 7890 曾长期维持约 256 个 `TIME_WAIT`。
- Python 后端私有工作集曾瞬时达到 282.5 MB，5 秒后回落到 74.1 MB，随后 25 秒到 78.4 MB；表现为可释放的周期分配峰值，不是单调泄漏，但会造成任务管理器瞬时占用偏高。

## 7. 发现的问题（按优先级）

### 已修复：灵动岛原生置顶层级错误

Windows 上的 `"floating"` 层级没有得到 `WS_EX_TOPMOST`，主窗口前置后会覆盖灵动岛。现已改用 `"screen-saver"`，并在所有主窗口层级变化路径上重新断言置顶。完整重启后，胶囊显示、展开、收起、最大化、F11 全屏和二次启动路径均通过；原生检查同时确认灵动岛未抢占主窗口焦点。

### 已修复：灵动岛媒体轮询短进程开销

折叠/无媒体轮询由 3 秒改为 15 秒，展开或播放为 5 秒；新增共享查询 Promise、控制串行链、5 秒查询超时、3 秒控制超时和隐藏暂停。20 秒实测仅产生 1 个媒体 PowerShell 子进程。

### 已修复：日报与通知导航协议

日报使用专用 `brief:show`，完整日报保留 `{ expanded: true }` payload，`calendar` 映射为 `memorial`。所有主窗口入口统一经过 renderer readiness 队列，崩溃 renderer 会 reload，发送失败不会丢弃待处理导航。

### P1：日历启动失败后不重试
