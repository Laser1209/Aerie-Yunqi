## Git pickaxe 候选只读复核
- 复核方式仅使用 `git diff-tree -G` 定位路径，并对对应 Git blob 运行脱敏的宽松/严格正则计数；未输出原始候选文本，未修改工作树或 Git 历史。
- `be9bb2ebb872`：`Spotlight/public/aerie-logo.svg` 的 7 个宽松命中全部位于内嵌图片 data URI；24 小时世界实施计划文档有 1 个宽松命中，字符统计为 153 字符、150 个小写字母、3 个连字符、0 个数字，并被更长模型标识/描述串包裹。两处严格边界命中均为 0。
- `f73948a9eef1`：`Aerie · 云栖.svg` 修改前后各有相同的 7 个宽松命中，全部位于同一内嵌图片 data URI；该 SVG 为单行大资源，整行变动使 Git pickaxe 选中提交，严格边界命中为 0。
- `40ce82b5562c`：首次加入同一 SVG 资源时出现相同 7 个 data URI 宽松命中，严格边界命中为 0。
- 裁决：`unconfirmed_git_pickaxe_commits=3` 是扫描器候选阶段的保守统计，不新增确认 finding；未来清洗范围只包含 16 条已确认历史 finding 所在的 4 个提交与 3 条路径。

## 推荐处置路线
1. 用户先在各 Provider 后台轮换/吊销任何可能对应的历史 key。
2. 在 clean working tree 上创建完整备份：
   - 记录 `git rev-parse HEAD`。
   - 复制整个仓库目录到外部备份位置。
   - 可选：导出 `git bundle create aerie-before-history-clean.bundle --all`。
3. 在临时克隆或备份副本中预演历史清洗，禁止直接在唯一工作副本上首次运行。
4. 清洗目标只限已确认的历史命中：
   - 删除或替换旧 `.env.example` / 旧规划文档中的 provider-key 形态值。
   - 替换 renderer 测试 fixture 历史中的 provider-key 形态假值。
   - 已确认 3 个 pickaxe 候选均为误报，不扩大清洗范围；预演时仍复核清洗后严格扫描为 0。
5. 清洗后重新运行验证：
   - `python tools\scan_provider_key_patterns.py`
   - `python tools\scan_provider_key_patterns.py --history`
   - `python -m pytest tests -q`
   - `node --test electron/tests/*.test.js`
   - `npm run check:all`（在 `electron` 目录）
   - Phase/Task 00–15 门禁脚本
6. 若全部通过，再更新 [[94_发布与安全检查]]，并将本文件 `status` 从 `pending_authorization` 改为 `completed`。
7. 若配置了 remote，需由用户确认是否 force push；若未配置 remote，记录本地历史已清理但无法上传。

- 快速预检：`python tools\history_remediation_preflight.py --no-history`。
- 完整预检：`python tools\history_remediation_preflight.py`。
- JSON 输出：`python tools\history_remediation_preflight.py --json`。
- 该工具只读取 Git 状态与脱敏扫描结果；不会执行 history rewrite、删除文件、轮换 key、push 或 force push。
- 工作区 clean 判断以暂存内容差异、未暂存内容差异和未跟踪文件为准，不依赖可能受 Windows CRLF/stat cache 影响的 `git status --porcelain`；真实 dirty entry 会按 `staged:`、`unstaged:`、`untracked:` 分类脱敏列出。
- 预期当前状态：存在真实未提交内容时报告 `clean=false`；提交后若历史未清理则报告 `history_findings=16`，并保持 `can_close=false`。

## 可接受的执行工具
- 首选：`git filter-repo`，只在用户授权后使用。
- 备选：BFG Repo-Cleaner，需先确认 Java/工具链可用。
- 禁止：未备份时直接修改当前唯一仓库历史；禁止未审计的全局字符串替换；禁止删除 `.git` 后重新初始化来伪装清理完成。

## 回滚与恢复
- 若 history rewrite 预演失败：丢弃临时克隆或从备份目录恢复。
- 若当前仓库已被重写但验证失败：使用预清洗 bundle 或备份目录恢复到原 `HEAD`。
- 若已推送到 remote 后发现问题：需与所有协作者协调重新 clone 或 reset；不得在未沟通时继续叠加 force push。

## 发布裁决选项
- `清洗并轮换`：执行 key 轮换、history rewrite、全量验证；通过后关闭发布门禁。
- `仅轮换不清洗`：记录风险接受，发布门禁不能标记为“未进入历史”，只能以例外裁决发布。
- `暂不处理`：保持发布阻塞，不宣称最终发布完成。
