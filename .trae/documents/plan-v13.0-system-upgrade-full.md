# Aerie v13.0 系统级功能升级实施方案

> [!abstract] 方案概览
> 本方案针对 Aerie · 云栖 AI 智能体系统的五大核心模块进行全面升级：屏幕控制与渲染修复、办公模式落地、回复质量管控、主动消息推送，以及通用质量保障。方案基于现有代码库深度调研，确保与 v12.x 架构完全兼容。

## 二、模块一：屏幕控制与渲染问题修复

### 2.1 屏幕控制模块深度修复

#### 2.1.1 问题根因

现有 `computer_control.py` 已实现完整的三档权限体系和操作方法，但：

1. **未注册到 `tool_registry`** — AI 无法通过 function call 调用
2. **`brain.py` 未传递 tools** — 对话时未启用工具调用模式
3. **前端审批流未接通** — 高风险操作需要用户确认的链路断裂

#### 2.1.2 修复方案

**步骤 1：工具注册 — 接入 tool_registry**

在 `pipeline.py` 初始化阶段，将 `computer_control` 的核心能力注册为工具：

| 工具名                 | 功能                    | 权限档    | 风险   |
| ---------------------- | ----------------------- | --------- | ------ |
| `screen_screenshot`  | 截图并返回描述          | VIEW_ONLY | safe   |
| `screen_window_info` | 获取当前窗口列表        | VIEW_ONLY | safe   |
| `screen_mouse_click` | 鼠标点击（坐标/元素）   | STANDARD  | medium |
| `screen_key_type`    | 键盘输入文字            | STANDARD  | medium |
| `screen_shell`       | 执行 shell 命令（受限） | STANDARD  | high   |
| `app_launch`         | 打开应用程序/文件       | STANDARD  | medium |
| `screen_uia_action`  | UIA 深度操控            | FULL      | high   |

**步骤 2：Brain 工具调用链路打通**

- 在 `brain.chat()` 中检测到工具调用意图时，自动切换到 `tools` 模式
- 实现 ReAct 循环：思考 → 调用工具 → 观察 → 回复
- 工具调用结果以「思考中...」状态展示给用户

**步骤 3：权限档位选择机制**

- 新增 `/api/permissions/level` 接口，查询/设置当前权限档
- 首次触发屏幕操作时，弹出档位选择弹窗（VIEW_ONLY / STANDARD / FULL）
- 选择结果持久化到 `settings.yaml`
- 高风险操作（shell、UIA）单独二次确认

**步骤 4：操作安全审查强化**

- 复用 `screen_action_sanitizer.py`，在工具执行前强制审查
- 危险命令黑名单 + 白名单双机制
- 所有操作写入审计日志（`data/audit/screen_audit.log`）

### 2.2 界面渲染异常修复

#### 2.2.1 消息渲染时序问题

**问题**：用户消息出现时，头像和昵称先显示，内容后加载，造成视觉跳动

**修复方案**：

1. 消息 DOM 一次性完整渲染（头像+昵称+内容占位符）
2. 流式输出时只更新 `.chat-msg__content` 区域，不重建整个消息节点
3. 头像预加载：启动时提前缓存 persona 和 user 的头像 dataURL

## 三、模块二：办公模式功能落地开发

### 3.1 办公模式架构设计

#### 3.1.1 模式切换机制

```
聊天模式 ←─────────────────────→ 办公模式
  (闲聊/情感)    上下文触发       (任务执行/工具调用)
       ↑                              ↑
       └──────── 无缝切换 ────────────┘
                (状态记忆 + 平滑过渡)
```

#### 3.1.2 办公模式核心能力

| 能力                     | 实现方式             | 说明                                               |
| ------------------------ | -------------------- | -------------------------------------------------- |
| **智能任务识别**   | 意图分类器           | 从对话中识别办公任务（写文档、查资料、整理数据等） |
| **上下文感知调用** | 技能路由 + 工具链    | 根据任务类型动态组合工具/技能                      |
| **多模型择优**     | provider_router 扩展 | 复杂任务用强模型，简单任务用快模型                 |
| **进度反馈**       | 流式状态更新         | 执行过程中实时反馈进度                             |
| **结果交付**       | 文档/表格/消息       | 根据任务类型选择最佳交付形式                       |

### 3.2 模型选型评估

基于 `.env` 中可用模型列表，评估办公场景适配度：

| 模型                | 推理能力   | 速度     | 成本       | 办公适配度 | 推荐场景           |
| ------------------- | ---------- | -------- | ---------- | ---------- | ------------------ |
| GPT-4o / Claude 3.5 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐   | ⭐⭐       | 95%        | 复杂推理、长文档   |
| 豆包 2.1            | ⭐⭐⭐⭐   | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 88%        | 中文办公、日常任务 |
| GLM-4               | ⭐⭐⭐⭐   | ⭐⭐⭐⭐ | ⭐⭐⭐⭐   | 85%        | 中文理解、代码     |
| Kimi                | ⭐⭐⭐⭐   | ⭐⭐⭐   | ⭐⭐⭐⭐   | 82%        | 长文档处理         |

**推荐策略**：默认豆包 2.1（成本最优+中文最强），复杂任务自动升级到 GPT-4o/Claude。

### 3.3 上下文感知式任务触发

**触发关键词/意图**：

- 📝 文档类：写报告、整理、总结、翻译、写邮件
- 📊 数据类：分析、统计、制表、对比
- 🔍 信息类：查资料、搜索、调研、对比方案
- ⚙️ 操作类：打开、设置、整理文件、截图处理
- 📅 日程类：安排、提醒、计划、预约

**触发流程**：

1. 用户消息 → 意图分类器
2. 识别为办公任务 → 静默切换到办公模式
3. 确认任务边界 → 执行计划生成
4. 工具/技能链执行 → 进度反馈
5. 结果交付 → 模式保持或回落

### 3.4 办公模式与聊天模式联动

- **无感切换**：用户无需手动切换，系统根据上下文自动判断
- **模式记忆**：记住当前模式，同一会话保持一致
- **平滑过渡**：切换时保持人设语气，不生硬
- **手动切换**：设置面板保留手动开关

### 3.5 设备识别能力

| 端                    | 识别方式              | 适配策略                                     |
| --------------------- | --------------------- | -------------------------------------------- |
| 💻 电脑端（Electron） | User-Agent + IPC 通道 | 完整功能：屏幕控制、文件操作、办公模式全能力 |
| 📱 手机端（QQ）       | 消息来源 + 设备信息   | 轻量办公：文本生成、信息查询、日程管理       |

- 新增 `message_source` 字段标记消息来源
- 不同设备触发不同的工具集和回复策略
- 手机端禁用屏幕操控类高风险功能

## 四、模块三：回复校验与质量管控优化

### 4.1 双层校验架构

```
AI 原始输出
    │
    ▼
┌─────────────────────┐
│  第一层：合规性校验  │  ← 事实准确性 + 安全合规
│  (Accuracy Guard)   │     + 人设一致性
└─────────┬───────────┘
          │ 通过
          ▼
┌─────────────────────┐
│  第二层：质量评判    │  ← 信息量评估 + 语气把控
│  (Quality Judge)    │     + 长度适中性
└─────────┬───────────┘
          │ 通过
          ▼
     最终输出给用户
```

### 4.2 第一层：合规性校验 (Accuracy Guard)

**校验维度**：

1. **事实准确性** — 检查是否有明显事实错误
2. **安全合规** — 不含危险/违规内容
3. **人设一致性** — 符合伊塔人设设定（温柔、恋人语气）
4. **功能权限认知** — 准确描述自身能力，不夸大不虚假

**实现方式**：

- 轻量级 LLM 调用（用快模型）做校验
- 系统提示词明确校验标准
- 校验失败时自动重写（最多 2 次），仍失败则降级输出

### 4.3 第二层：质量评判 (Quality Judge)

**评判维度**：

| 维度     | 评分标准                       | 权重 |
| -------- | ------------------------------ | ---- |
| 信息量   | 是否足够回答问题，不冗余不敷衍 | 40%  |
| 语气贴合 | 人设是否一致，恋人感是否到位   | 30%  |
| 长度适中 | 根据问题复杂度匹配长度         | 20%  |
| 情感价值 | 有没有温度，会不会关心人       | 10%  |

**执行策略**：

- 优先级：**准确性 > 速度** — 宁可慢 1s，也要保证质量
- 简单问句（聊天/问候）跳过质检，保证响应速度
- 复杂问题/长回答强制质检
- 质检结果写入日志，用于后续模型微调参考

### 4.4 接入位置

```
pipeline.py
  └── chat()
       ├── brain.chat()           ← 生成原始回复
       ├── accuracy_guard.check() ← 第一层校验（新增）
       ├── quality_judge.score()  ← 第二层评判（新增）
       └── return final_output
```

```

│   ├── __init__.py
│   ├── accuracy_guard.py      # 合规性校验
```

## 五、模块四：主动消息推送能力开发

### 5.1 现有基础盘点

✅ **已实现**：

- `push_scheduler.py` — Cron 调度器
- `proactive_judge.py` — 推送时机判定
- `PushPolicy` — 频率限制 + 静默时段
- 6 个预置场景（早安、简报、闲聊、思念、启动问候等）
- QQ 通道（NapCat）已打通

❌ **待增强**：

- 缺少「主动发起对话」能力（不仅是推送通知）
- 缺少 Electron 端主动弹窗/动态岛推送
- 缺少事件驱动型主动消息（基于用户行为触发）
- 缺少对话式主动互动（不是单向通知，而是开启话题）

### 5.2 开源方案调研

| 方案                       | 实现思路                         | 适配度     | 优点               | 缺点             |
| -------------------------- | -------------------------------- | ---------- | ------------------ | ---------------- |
| **CRON + LLM 生成**  | 定时触发 + 大模型生成文案        | ⭐⭐⭐⭐⭐ | 实现简单，灵活度高 | 依赖调度准确性   |
| **状态机驱动**       | 基于用户状态 + 时间 + 事件触发   | ⭐⭐⭐⭐   | 智能度高，更像真人 | 状态设计复杂     |
| **情感引擎联动**     | 结合情感/思念值触发主动联系      | ⭐⭐⭐⭐⭐ | 符合恋人设，体验好 | 需要情感数据积累 |
| **Webhook 事件驱动** | 监听外部事件（日历、天气等）触发 | ⭐⭐⭐     | 场景丰富           | 需要对接更多 API |

**推荐方案**：**情感引擎 + CRON + 事件驱动 三位一体**

## 六、通用落地实施要求

### 6.1 实施阶段划分

| 阶段              | 内容                               | 预计工作量 | 验收标准                    |
| ----------------- | ---------------------------------- | ---------- | --------------------------- |
| **Phase 1** | 屏幕控制打通 + 渲染修复 + 权限流程 | 高         | AI 能实际操作电脑，授权正常 |
| **Phase 2** | 办公模式落地 + 设备识别            | 高         | 能识别办公意图并执行任务    |
