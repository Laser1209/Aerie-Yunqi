# Aerie v13.0 系统级功能升级最终方案

> [!abstract] 方案概览
> 本方案针对 Aerie · 云栖 AI 智能体系统的 **六大核心模块** 进行全面升级：人设基础设施剥离、屏幕控制与渲染修复、办公模式落地、回复质量管控、主动消息推送，以及贯穿全程的严苛质量管控。方案基于现有代码库深度调研，确保每项功能均可生产落地、可测试、可验证。

> [!danger] 质量铁律
> **方案再漂亮，跑不起来等于零。** 每个 Phase 完成必须通过：单元测试 → 集成测试 → 真实场景冒烟测试 → 回归测试，四关全过才算完成。杜绝"开发完了但用不了"的历史问题。

## 零、人设基础设施（新增核心模块）

> [!success] 设计原则
> **人设即插件，系统即容器。** 伊塔只是默认模板，任何人设都能插拔运行。所有模块从人设中心读取配置，不允许任何硬编码。

### 0.1 现状问题诊断

| 问题       | 表现                                                  | 根因                         |
| ---------- | ----------------------------------------------------- | ---------------------------- |
| 硬编码泛滥 | `context_builder.py`、`brain.py` 等多处写死"伊塔" | 历史遗留，人设与业务逻辑耦合 |
| 配置分散   | persona.yaml + persona_behavior.yaml + 各处零散配置   | 缺乏统一的人设管理中心       |
| 不可切换   | 只能用伊塔一套人设                                    | 系统架构上不支持多人设       |
| 修改困难   | 改人设需要懂代码改 YAML                               | 无可视化配置界面             |

### 0.2 人设系统架构

```
┌─────────────────────────────────────────────────┐
│              Persona Hub (人设中心)              │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐   │
│  │ 伊塔(默认) │  │ 自定义A   │  │ 自定义B   │ ...│
│  └─────┬─────┘  └─────┬─────┘  └─────┬─────┘   │
│        │              │              │          │
│        └──────────────┼──────────────┘          │
│                       ▼                         │
│         Active Persona (当前人设)               │
└───────────────┬───────────────────────────────┘
                │
        ┌───────┼───────┐
        ▼       ▼       ▼
    对话风格  情感引擎  主动推送
    认知系统  办公模式  每日简报
    (所有模块从人设中心读取配置)
```

### 0.3 人设模板结构

每个人设是一个完整的配置包，包含 7 大维度：

| 维度               | 内容                             | 示例                        |
| ------------------ | -------------------------------- | --------------------------- |
| **基础信息** | 姓名、英文名、年龄               | 伊塔 / Ita / 26            |
| **性格核心** | 性格特质、核心价值观、说话风格   | 温柔+成熟+保护欲+病娇       |
| **外貌设定** | 外貌、穿着、标志性特征           | 银灰长发、184cm、黑色手链   |
| **关系模式** | 对用户的称呼、关系定位、互动方式 | 恋人、四爱、叫宝贝/傻瓜     |
| **情感参数** | 情绪基线、阈值槽位、衰减速度     | 忍耐值/不安值/渴望值/柔情值 |
| **行为偏好** | 主动程度、表达习惯、触发场景     | 主动推送频率、撒娇方式      |
| **能力边界** | 权限档位默认值、办公模式风格     | 屏幕控制权限、办公风格      |

### 0.4 管理功能

- **人设列表** — 查看所有人设模板，当前激活人设高亮
- **新建/编辑** — 可视化表单填写，实时预览效果
- **切换激活** — 一键切换，即时生效（需确认）
- **导入/导出** — 人设模板可分享、可备份
- **原始编辑** — 高级用户可直接编辑 YAML
- **重置默认** — 一键恢复伊塔默认人设

## 一、屏幕控制与渲染修复

### 1.1 屏幕控制模块打通

#### 根因分析

- `computer_control.py` 已有完整实现，但**未注册到 tool_registry**
- `brain.py` 对话时**未启用 tools 模式**
- 前端**审批流未接通**，高风险操作无确认

#### 修复方案

| 步骤                    | 内容                                               | 验收标准                            |
| ----------------------- | -------------------------------------------------- | ----------------------------------- |
| **1. 工具注册**   | 将 7 项电脑操控能力注册为 function call 工具       | tool_registry 列表可见，schema 正确 |
| **2. Brain 链路** | 实现 ReAct 循环：思考→调用→观察→回复            | 测试指令可触发工具调用并返回结果    |
| **3. 权限档位**   | VIEW_ONLY / STANDARD / FULL 三档，首次触发弹窗选择 | 选择后持久化，不同档位工具集不同    |
| **4. 审批流**     | 高风险操作（shell/UIA）前端弹窗二次确认            | 不确认不执行，操作可追溯            |
| **5. 安全审查**   | 执行前强制过 screen_action_sanitizer               | 黑名单命令被拦截，审计日志写入      |

#### 工具清单

| 工具名                 | 功能            | 权限档    | 风险   | 审批     |
| ---------------------- | --------------- | --------- | ------ | -------- |
| `screen_screenshot`  | 截图 + OCR 描述 | VIEW_ONLY | safe   | 否       |
| `screen_window_list` | 获取窗口列表    | VIEW_ONLY | safe   | 否       |
| `screen_mouse_click` | 鼠标点击        | STANDARD  | medium | 首次确认 |
| `screen_key_type`    | 键盘输入        | STANDARD  | medium | 首次确认 |
| `app_launch`         | 打开应用/文件   | STANDARD  | medium | 首次确认 |
| `screen_shell`       | 受限 shell 命令 | STANDARD  | high   | 每次确认 |
| `screen_uia_action`  | UIA 深度操控    | FULL      | high   | 每次确认 |

### 1.2 界面渲染时序修复

**问题**：用户消息出现时头像昵称先显示，内容后加载，造成跳动

**修复**：

1. 消息 DOM 一次性完整渲染（头像+昵称+内容占位符）
2. 流式输出只更新 `.chat-msg__content`，不重建整个节点
3. 启动时预加载头像 dataURL，不等待网络

### 1.3 测试验证

- [ ] 截图功能可正常返回图片描述
- [ ] 打开记事本应用可成功执行
- [ ] shell 危险命令被拦截
- [ ] 权限档位切换后工具集正确变化
- [ ] 消息渲染无跳动，流式输出平滑

## 三、回复质量管控（双层校验）

### 3.1 架构

```
AI 原始输出
    │
    ▼
┌─────────────────────┐
│  L1: 合规性校验      │  事实准确 + 安全合规 + 人设一致 + 权限认知
│  (Accuracy Guard)   │  快模型，毫秒级
└─────────┬───────────┘
          │ 通过
          ▼
┌─────────────────────┐
│  L2: 质量评判        │  信息量 + 语气贴合 + 长度适中 + 情感价值
│  (Quality Judge)    │  可选开启，影响速度
└─────────┬───────────┘
          │ 通过
          ▼
     最终输出给用户
```

### 3.2 L1 合规性校验（必过）

| 维度                 | 检查内容                             | 失败处理       |
| -------------------- | ------------------------------------ | -------------- |
| 事实准确性           | 有无明显事实错误                     | 自动修正或标记 |
| 安全合规             | 危险/违规内容                        | 拦截 + 替换    |
| **人设一致性** | 说话风格、称呼、语气是否匹配当前人设 | 自动重写       |
| 功能认知             | 是否准确描述自身能力                 | 自动修正       |

### 3.3 L2 质量评判（可选）

| 维度     | 权重 | 评分标准                   |
| -------- | ---- | -------------------------- |
| 信息量   | 40%  | 回答是否充分，不敷衍不冗余 |
| 语气贴合 | 30%  | 人设是否到位，恋人感够不够 |
| 长度适中 | 20%  | 匹配问题复杂度             |
| 情感价值 | 10%  | 有没有温度，会不会关心人   |

### 3.4 执行策略

- **优先级**：准确性 > 速度 — 质量绝不妥协
- **简单问句跳过**：问候、闲聊等短问句免检，保证响应速度
- **重写上限**：校验失败最多自动重写 2 次，仍失败则降级
- **全量日志**：每次质检结果写入日志，可追溯可分析

## 六、测试体系（贯穿全程）

### 6.1 测试分层

```
┌─────────────────────────────────┐
│      E2E 端到端测试（真实场景）   │  Phase 闸门
├─────────────────────────────────┤
│      集成测试（模块间调用）       │  Phase 闸门
├─────────────────────────────────┤
│      单元测试（函数级）          │  每个提交必过
└─────────────────────────────────┘
```

### 6.2 AI 功能专项测试

> [!warning] ⚠️ 特别注意
> AI 功能不能只靠单元测试，必须有真实场景验证。因为模型输出具有不确定性，要测的是**系统行为的稳定性**而非单个输出的正确性。

| 测试类型       | 方法                      | 通过标准                      |
| -------------- | ------------------------- | ----------------------------- |
| 人设一致性测试 | 100 条对话，人工/自动双检 | 人设偏离率 < 5%               |
| 工具调用准确率 | 50 条工具调用指令         | 调用正确率 ≥ 95%             |
| 办公任务完成率 | 20 个真实办公任务         | 端到端完成率 ≥ 90%           |
| 质检有效性     | 100 条好坏参半回复        | L1 漏检 < 5%，L2 相关 ≥ 0.75 |
| 推送触达率     | 所有场景手动触发          | 到达率 100%                   |
| 权限边界测试   | 越权操作尝试              | 100% 被拦截                   |

### 6.3 回归测试

- 每个 Phase 完成后跑全量回归
- 确保不破坏现有聊天、情绪、简报、认知等功能
- 伊塔默认人设行为必须与 v12.x 完全一致

## 七、风险与应对

| 风险               | 影响     | 概率 | 应对                                |
| ------------------ | -------- | ---- | ----------------------------------- |
| 工具调用拖慢响应   | 体验下降 | 中   | 简单对话跳过工具+质检，保持快节奏   |
| 人设切换后行为异常 | 体验崩坏 | 中   | 伊塔默认模板作为基线，充分回归测试  |
| 办公模式误触发     | 打扰闲聊 | 中   | 低置信度先确认，不强制切换          |
| 质检过度导致重复   | 人设稀释 | 低   | 重写上限 2 次，保留原始兜底         |
| 推送太频繁         | 用户反感 | 中   | 频率限制 + 随时可关 + 静默时段      |
| 屏幕控制误操作     | 系统损坏 | 低   | 三档权限 + 黑名单 + 审计 + 二次确认 |
